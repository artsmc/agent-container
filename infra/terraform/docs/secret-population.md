# Secret Population Guide

After `terraform apply` completes, the Secret Manager secret containers exist but contain no values. This guide explains how to populate each of the 13 application secrets.

## Overview

The `secrets` module creates placeholder secrets with no versions. Application services cannot start until their required secrets are populated. The CI/CD service account has `secretVersionManager` access on all secrets and should be used for automated population.

## Two Types of Database Secrets

There are two sets of database-related secrets:

1. **Auto-generated (database/auth-database modules)**: Created by Terraform with the actual connection strings. Secret IDs are `{env}-db-url` and `{env}-auth-db-url`. These are used by ops/bootstrap tooling.

2. **Application secrets (secrets module)**: Placeholders named `{project}-{env}-DATABASE_URL` and `{project}-{env}-AUTH_DATABASE_URL`. These are what application services actually use (IAM bindings from the `iam` module point here). Populate these from the auto-generated secrets.

## Step 1: Get the Auto-Generated Database URLs

```bash
ENV="dev"
PROJECT="iexcel-dev"

# Product database URL (auto-generated by database module)
DB_SECRET=$(terraform output -raw database_connection_secret)
DATABASE_URL=$(gcloud secrets versions access latest \
  --secret="${DB_SECRET}" \
  --project="${PROJECT}")

# Auth database URL (auto-generated by auth-database module)
AUTH_DB_SECRET=$(terraform output -raw auth_database_connection_secret)
AUTH_DATABASE_URL=$(gcloud secrets versions access latest \
  --secret="${AUTH_DB_SECRET}" \
  --project="${PROJECT}")

echo "DATABASE_URL: ${DATABASE_URL}"
echo "AUTH_DATABASE_URL: ${AUTH_DATABASE_URL}"
```

## Step 2: Populate DATABASE_URL and AUTH_DATABASE_URL

```bash
PROJECT_NAME="iexcel"
ENV="dev"
PROJECT="iexcel-dev"

# Populate DATABASE_URL
echo -n "${DATABASE_URL}" | gcloud secrets versions add \
  "${PROJECT_NAME}-${ENV}-DATABASE_URL" \
  --data-file=- \
  --project="${PROJECT}"

# Populate AUTH_DATABASE_URL
echo -n "${AUTH_DATABASE_URL}" | gcloud secrets versions add \
  "${PROJECT_NAME}-${ENV}-AUTH_DATABASE_URL" \
  --data-file=- \
  --project="${PROJECT}"
```

## Step 3: Generate OIDC Signing Keys

The auth service uses RS256 JWT signing. Generate a 2048-bit RSA key pair:

```bash
# Generate private key (PEM format)
openssl genrsa -out signing_key.pem 2048

# Extract public key
openssl rsa -in signing_key.pem -pubout -out signing_key_pub.pem

# Populate SIGNING_KEY_PRIVATE
gcloud secrets versions add \
  "${PROJECT_NAME}-${ENV}-SIGNING_KEY_PRIVATE" \
  --data-file=signing_key.pem \
  --project="${PROJECT}"

# Populate SIGNING_KEY_PUBLIC
gcloud secrets versions add \
  "${PROJECT_NAME}-${ENV}-SIGNING_KEY_PUBLIC" \
  --data-file=signing_key_pub.pem \
  --project="${PROJECT}"

# Delete local key files after population
shred -u signing_key.pem signing_key_pub.pem
```

## Step 4: Configure External Identity Provider

If using an external OIDC/OAuth2 provider for federated login, obtain the client credentials from your IdP dashboard:

```bash
# IDP_CLIENT_ID — from your OAuth app registration
echo -n "your-idp-client-id" | gcloud secrets versions add \
  "${PROJECT_NAME}-${ENV}-IDP_CLIENT_ID" \
  --data-file=- \
  --project="${PROJECT}"

# IDP_CLIENT_SECRET — from your OAuth app registration
echo -n "your-idp-client-secret" | gcloud secrets versions add \
  "${PROJECT_NAME}-${ENV}-IDP_CLIENT_SECRET" \
  --data-file=- \
  --project="${PROJECT}"
```

### Google Service Account

The `GOOGLE_SERVICE_ACCOUNT_JSON` secret holds a Google Workspace service account key for accessing Google APIs (Docs, Sheets, Calendar, etc.):

```bash
# Generate and download the service account key from GCP Console or:
gcloud iam service-accounts keys create google-sa-key.json \
  --iam-account="workspace-access@${PROJECT}.iam.gserviceaccount.com" \
  --project="${PROJECT}"

gcloud secrets versions add \
  "${PROJECT_NAME}-${ENV}-GOOGLE_SERVICE_ACCOUNT_JSON" \
  --data-file=google-sa-key.json \
  --project="${PROJECT}"

shred -u google-sa-key.json
```

## Step 6: Verify All Secrets Are Populated

Use this script to check which secrets have at least one version:

```bash
PROJECT_NAME="iexcel"
ENV="dev"
PROJECT="iexcel-dev"

SECRETS=(
  "DATABASE_URL"
  "AUTH_DATABASE_URL"
  "IDP_CLIENT_ID"
  "IDP_CLIENT_SECRET"
  "SIGNING_KEY_PRIVATE"
  "SIGNING_KEY_PUBLIC"
  "ASANA_CLIENT_ID"
  "ASANA_CLIENT_SECRET"
  "ASANA_ACCESS_TOKEN"
  "GRAIN_API_KEY"
  "GOOGLE_SERVICE_ACCOUNT_JSON"
  "LLM_API_KEY"
  "EMAIL_PROVIDER_API_KEY"
)

echo "Checking secret population status for ${ENV}..."
for SECRET in "${SECRETS[@]}"; do
  SECRET_ID="${PROJECT_NAME}-${ENV}-${SECRET}"
  VERSION_COUNT=$(gcloud secrets versions list "${SECRET_ID}" \
    --project="${PROJECT}" \
    --filter="state=ENABLED" \
    --format="value(name)" 2>/dev/null | wc -l)
  if [ "${VERSION_COUNT}" -gt 0 ]; then
    echo "[OK]      ${SECRET_ID}"
  else
    echo "[MISSING] ${SECRET_ID}"
  fi
done
```

## Secret Rotation

To rotate a secret (e.g., after a credential leak):

```bash
# Add a new version (this becomes the latest)
echo -n "new-secret-value" | gcloud secrets versions add \
  "${PROJECT_NAME}-${ENV}-LLM_API_KEY" \
  --data-file=- --project="${PROJECT}"

# Disable the old version
gcloud secrets versions disable 1 \
  --secret="${PROJECT_NAME}-${ENV}-LLM_API_KEY" \
  --project="${PROJECT}"
```

Cloud Run services pick up the new version automatically on the next request (Secret Manager injection caches for ~1 minute). A service restart forces immediate pickup:

```bash
gcloud run services update SERVICE_NAME \
  --region=REGION \
  --project="${PROJECT}"
```

## Security Best Practices

- Never store secret values in source control, CI/CD logs, or shell history
- Use `--data-file=-` to pipe values via stdin rather than passing on the command line
- Run `shred -u` on any temporary key files immediately after population
- Enable Secret Manager audit logging to track all access: the Terraform config automatically enables this via the project-level audit log configuration
- For production, require manager approval before accessing secrets with a custom access policy
