# Feature Requirements Document
# Feature 20: Workflow B — Agenda Agent

**Feature Name:** workflow-b-agenda-agent
**Phase:** Phase 3 — External Integrations & Workflows
**Date:** 2026-03-03
**Status:** Specification

---

## 1. Business Objective

Before each client follow-up call, iExcel account managers manually compile a "Running Notes" document that summarizes completed tasks, highlights deliverables, and sets the agenda for the upcoming call. This document is central to the client relationship — it demonstrates what has been accomplished, surfaces recommendations, and proposes next steps. Currently, it requires the account manager to manually review Asana, pull completed tasks, synthesize them into readable summaries, and write the full document.

Feature 20 automates this pre-call document generation by implementing the Mastra agent responsible for Workflow B (Pre-Call to Build Agenda). The agent receives reconciled task data (Postgres metadata merged with live Asana status from Feature 13), generates a structured Running Notes document in the six-section format, and saves it as a draft agenda via the API layer. Human review remains required before the document is shared with the client.

---

## 2. Problem Being Solved

The manual Running Notes process has three core failure modes:

1. **Time cost.** Compiling a Running Notes document requires the account manager to cross-reference Asana, understand what was completed in the cycle, and write human-readable summaries — a process that can take 30 minutes to an hour per client, per cycle.
2. **Inconsistency.** The quality and structure of Running Notes varies by account manager and by the time available before the call. Clients experience uneven presentation of their work status.
3. **Raw data presentation.** When compiled manually under time pressure, Running Notes can default to a task-by-task data dump rather than theme-grouped summaries that tell a coherent story about the work completed.

---

## 3. Feature Description

The agenda agent is a Mastra agent that implements one focused capability: given a set of reconciled tasks for a client (tasks that have been pushed to Asana and have live completion status merged back), produce a Running Notes document in the standard six-section format and persist it as a draft agenda via the API.

The agent operates within these boundaries:
- It receives reconciled task data from the workflow orchestration layer (Feature 17), which has already invoked Feature 13 (Status Reconciliation) before calling this agent.
- It classifies tasks as completed or incomplete based on the `asanaStatus` field from the reconciled data.
- It calls `POST /clients/{id}/agendas` through the `api-client` package to save the draft agenda.
- It returns the created agenda short ID to the workflow orchestration layer.
- It does not call Asana directly. It does not finalize, share, or email agendas.

## 4. User Stories

### US-01: Account Manager Initiates Agenda Workflow
As an account manager preparing for a client follow-up call, I want to trigger the agenda workflow so that a draft Running Notes document is automatically generated from completed Asana tasks without manual compilation.

**Acceptance Criteria:**
- The account manager provides a `client_id` and optional `cycle_start`/`cycle_end` to the workflow trigger endpoint.
- The system verifies that completed tasks exist before invoking the agent.
- A draft agenda appears in the agenda management interface with `status = draft`.

### US-02: Theme-Based Task Summaries
As a client reviewing the Running Notes, I want completed work to be presented in thematic groupings rather than as a raw task list so that I can understand what was accomplished at a project level rather than reading individual ticket titles.

**Acceptance Criteria:**
- The Completed Tasks section groups tasks by theme or project (not one task per line).
- Each theme group has a human-readable summary written in prose.
- The summaries read as if written by a project manager, not generated by pulling task titles.

### US-03: All Six Running Notes Sections Present
As an account manager reviewing the draft, I want the Running Notes document to always contain all six standard sections so that I have a consistent structure to edit and finalize before the call.

**Acceptance Criteria:**
- The generated agenda contains all six sections: Completed Tasks, Incomplete Tasks, Relevant Deliverables, Recommendations, New Ideas, Next Steps.
- No section is absent, even if the content is brief.
- Data-driven sections (Completed Tasks, Incomplete Tasks) are based on the reconciled task data.
- Advisory sections (Recommendations, New Ideas, Next Steps) are generated by the agent based on task patterns and context.

### US-04: No Empty Agendas
As a system operator, I want the workflow to refuse to generate an agenda when no completed tasks exist so that account managers are not handed an empty or meaningless document before a client call.

**Acceptance Criteria:**
- If the reconciled task data contains zero tasks with `asanaStatus = 'completed'`, the workflow returns a `422 NO_COMPLETED_TASKS` response (handled by Feature 17, before this agent is invoked).
- The agent itself also guards against an empty completed-task set and returns a meaningful warning if the completed tasks count drops to zero after filtering.
- The workflow run is never marked `completed` with an empty agenda.

### US-05: Cycle Date Range in Saved Agenda
As an account manager using the agenda management interface, I want each saved agenda to carry the cycle start and end dates so that I can understand which work period it covers.

**Acceptance Criteria:**
- The saved agenda record includes `cycleStart` and `cycleEnd` as ISO 8601 date strings.
- These dates come from the workflow invocation context (provided by Feature 17).

### US-06: Draft Status for Human Review
As an account manager, I want the agent-generated agenda to be saved as a draft so that I can review, edit, and finalize it before it is shared with the client.

**Acceptance Criteria:**
- All agent-generated agendas are created with `status = draft`.
- The agenda cannot be shared or emailed until it has been finalized (enforced by the API layer, Feature 14).

### US-07: Client Scoping
As a system, all agendas generated by the agent must be scoped to the triggering client so that cross-client data leakage is impossible.

**Acceptance Criteria:**
- The generated agenda carries the `client_id` from the workflow invocation context.
- Reconciled task data is pre-filtered to the specified client before being passed to the agent.
- The agent does not combine task data from multiple clients.

## 5. Success Metrics

| Metric | Target |
|---|---|
| Agenda section completeness | 100% of generated agendas have all six sections populated |
| Theme grouping quality | Account manager edits Completed Tasks section in fewer than 50% of reviews (baseline — measured post-launch) |
| Workflow completion rate | Less than 2% of agenda workflow runs fail due to agent error |
| Time savings | Average pre-call preparation time reduced from estimated 30–60 minutes to under 10 minutes for review/edit |
| Human edit rate | Average less than 3 significant edits per agenda before finalization (baseline — measured post-launch) |

---

## 6. Business Constraints

- The agent uses reconciled data from Feature 13 — it never calls Asana directly.
- Agendas are created as `status = draft`. Sharing and emailing require finalization (Feature 14).
- The agent runs on the Mastra runtime established in Feature 18.
- Workflow B can be manually triggered or scheduled, but the scheduling mechanism is outside this feature's scope.
- If no completed tasks exist, the workflow does not proceed to agent invocation (Feature 17 enforces this guardrail before invoking the agent). However, the agent itself must also validate this condition defensively.

---

## 7. Running Notes Format

The six-section format is established in `asana-call-agenda.md` and has been used by iExcel for client communication. The agent must produce content matching this structure:

| Section | Content Source | Notes |
|---|---|---|
| **Completed Tasks** | Data-driven: reconciled tasks with `asanaStatus = 'completed'` | Grouped by theme/project, written as prose summaries — not a task list |
| **Incomplete Tasks** | Data-driven: reconciled tasks with `asanaStatus = 'incomplete'` | Presented with context on why they are outstanding, if inferable |
| **Relevant Deliverables** | Agent-generated: based on completed work output | Links completed tasks to tangible deliverables or artifacts |
| **Recommendations** | Agent-generated: based on task patterns and client context | Forward-looking suggestions based on what the work revealed |
| **New Ideas** | Agent-generated: based on task context and emerging themes | Suggestions that surfaced from the work completed this cycle |
| **Next Steps** | Agent-generated: forward-looking action items | Sets the agenda for the next cycle |

---

## 8. Out of Scope

- Status reconciliation with Asana (Feature 13 — agent receives pre-reconciled data)
- Agenda lifecycle management: finalization, sharing, emailing (Feature 14)
- Google Docs export (Feature 15 — triggered after agenda review)
- Direct Asana API calls (all Asana data arrives via reconciled task input)
- Workflow triggering and orchestration (Feature 17)
- Mastra runtime setup (Feature 18)
- Scheduling automation for Workflow B triggers
