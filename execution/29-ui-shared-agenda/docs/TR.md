## 3. API Contract

### Endpoint Used

```
GET /shared/{token}
```

**Authentication:** None — public endpoint, no `Authorization` header.

**Request:**
- Method: `GET`
- Path param: `token` — opaque string generated by `POST /agendas/{id}/share`
- No query params, no request body

**Success Response (`200 OK`):**
```typescript
interface SharedAgendaResponse {
  agenda_id: string
  short_id: string              // "AGD-0015"
  client_name: string           // "Acme Corp"
  cycle_start: string           // "2026-02-01" (ISO date)
  cycle_end: string             // "2026-02-28" (ISO date)
  finalized_at: string          // "2026-02-28T14:30:00Z" (ISO datetime)
  running_notes: {
    completed_tasks: string     // Rich text content (format: TBD — markdown or HTML)
    incomplete_tasks: string
    relevant_deliverables: string
    recommendations: string
    new_ideas: string
    next_steps: string
  }
  token_expires_at: string | null  // ISO datetime or null (no expiry)
}
```

**Error Responses:**
| Status | Meaning | UI Action |
|---|---|---|
| `404 Not Found` | Token does not exist or was revoked | Render `InvalidLinkError` |
| `410 Gone` | Token has passed expiry | Render `ExpiredLinkError` |
| `403 Forbidden` | Agenda not finalized | Render `GenericError` |
| `5xx` / network | Server error | Render `GenericError` |

**Implementation note:** The `api-client` package's `shared.getByToken(token)` method must be called from within the Next.js Server Component `page.tsx`. The method throws typed errors that can be caught and inspected for status codes.

## 4. Component Specifications

### 4.1 Page Component (`app/shared/[token]/page.tsx`)

```typescript
// Next.js 16 App Router — params is a Promise
export default async function SharedAgendaPage({
  params,
}: {
  params: Promise<{ token: string }>
}) {
  const { token } = await params

  let agenda: SharedAgendaResponse

  try {
    agenda = await apiClient.shared.getByToken(token)
  } catch (error) {
    if (isApiError(error, 404)) {
      return <SharedAgendaError type="invalid" />
    }
    if (isApiError(error, 410)) {
      return <SharedAgendaError type="expired" />
    }
    return <SharedAgendaError type="generic" />
  }

  return (
    <PublicLayout>
      <AgendaHeader agenda={agenda} />
      <RunningNotesViewer runningNotes={agenda.running_notes} />
      <PrintActions />
    </PublicLayout>
  )
}
```

**Key decisions:**
- `params` is awaited per Next.js 16 API (params is a `Promise<{ token: string }>`)
- Error handling is done via try/catch in the Server Component, not middleware
- Each error type maps to a distinct error variant in `SharedAgendaError`
- `PublicLayout` wraps the full content

### 4.2 `generateMetadata` Function

```typescript
export async function generateMetadata({
  params,
}: {
  params: Promise<{ token: string }>
}): Promise<Metadata> {
  const { token } = await params

  try {
    const agenda = await apiClient.shared.getByToken(token)
    return {
      title: `${agenda.client_name} — Agenda ${agenda.short_id} | iExcel`,
      description: `Shared agenda for ${agenda.client_name}, covering ${formatDateRange(agenda.cycle_start, agenda.cycle_end)}.`,
      robots: { index: false, follow: false },
    }
  } catch {
    return {
      title: 'Agenda | iExcel',
      robots: { index: false, follow: false },
    }
  }
}
```

**Key decisions:**
- Next.js 16 `generateMetadata` co-located in the same `page.tsx` file
- Data fetch is deduplicated by React's cache — `getByToken(token)` called in both `generateMetadata` and the page component will not result in two API calls (React.cache memoization applies within a single request)
- `robots: { index: false, follow: false }` on all shared agenda pages — these are private client documents

### 4.5 `RunningNotesSection` Component

```typescript
// Server Component
interface RunningNotesSectionProps {
  heading: string
  content: string | null | undefined
}

const EMPTY_PLACEHOLDER = 'Nothing to report for this period.'

export function RunningNotesSection({ heading, content }: RunningNotesSectionProps) {
  const isEmpty = !content || content.trim() === ''
  return (
    <section className={styles.section}>
      <h2 className={styles.sectionHeading}>{heading}</h2>
      <div className={styles.sectionContent}>
        {isEmpty
          ? <p className={styles.emptyPlaceholder}>{EMPTY_PLACEHOLDER}</p>
          : <RichTextRenderer content={content} />
        }
      </div>
    </section>
  )
}
```

**Rich text rendering:** A `RichTextRenderer` utility component handles content display. Implementation options:
- If content is **markdown**: use a lightweight, SSR-safe renderer (e.g. `marked` or `remark` with `remark-html`), output sanitized with `DOMPurify` equivalent (server-safe: `sanitize-html`)
- If content is **HTML**: output sanitized HTML using `sanitize-html` with a whitelist of safe tags (`p`, `ul`, `ol`, `li`, `strong`, `em`, `br`, `h3`, `h4`)

The exact format of `running_notes` content strings is determined by the agenda editor (Feature 28) and must be confirmed during integration. Both paths must be supported or the format locked down before implementation.

## 6. Data Validation and Sanitization

### 6.1 Token Validation

The token from the URL param is passed directly to the API. No client-side validation of token format is needed — the API is the authority on token validity.

The token must be passed as-is: `const { token } = await params` — no transformation.

### 6.2 Content Sanitization

Running Notes content is user-generated rich text from the agenda editor. Before rendering as HTML in the browser:

- If rendered as HTML: sanitize using `sanitize-html` with an explicit allowlist
- Allowed tags: `p`, `ul`, `ol`, `li`, `strong`, `em`, `br`, `h3`, `h4`, `a` (href restricted to safe protocols)
- Disallowed: `script`, `style`, `iframe`, `object`, `embed`, `form`, `input`
- Attributes: only `href` on `<a>` tags (enforce `rel="noopener noreferrer"`)

This sanitization must run server-side (in the Server Component), not client-side.

## 7. Performance Requirements

| Metric | Target | Approach |
|---|---|---|
| Time to First Byte (TTFB) | < 500ms | Server Component renders synchronously after single API call |
| First Contentful Paint (FCP) | < 1.5s | SSR delivers full HTML — no client hydration delay for content |
| Largest Contentful Paint (LCP) | < 2.5s | Content is text-based — no large image assets |
| JavaScript bundle (client) | < 10KB | Only `PrintActions` ships client JS |
| Cumulative Layout Shift (CLS) | < 0.1 | No dynamic layout shifts — content rendered server-side |

### Caching

The `/shared/{token}` API endpoint returns a finalized agenda — the content does not change after finalization. However, for V1, do not apply aggressive caching strategies on the Next.js side. The API layer controls caching headers. The Server Component should not add `cache: 'force-cache'` or revalidation time — defer to API cache-control headers.

Reason: token expiry and revocation must reflect immediately. A cached page for a revoked token would be a data security issue.

## 8. Security Considerations

### 8.1 Token is the Access Credential

The token in the URL is the sole access control mechanism. Security guarantees:
- Tokens are generated by the API with sufficient entropy (API responsibility)
- Tokens are scoped to a single agenda and client (API responsibility)
- The UI passes the token directly to the API and renders the response — it has no role in validating token legitimacy

### 8.2 No Auth Middleware

The `/shared/*` route group must be explicitly excluded from any auth middleware configured in `middleware.ts`. This must be verified when the auth middleware is implemented in Feature 24.

```typescript
// middleware.ts (from Feature 24 — this feature must confirm exclusion)
export const config = {
  matcher: [
    // Exclude /shared/* from auth middleware
    '/((?!shared|_next/static|_next/image|favicon.ico).*)',
  ],
}
```

### 8.3 No Internal Data Leakage

The API's `GET /shared/{token}` response must not include internal fields. The UI must not attempt to render any fields beyond those in the `SharedAgendaResponse` interface. TypeScript strict mode enforces this at compile time.

### 8.4 XSS Prevention

Rich text content rendered as HTML must always pass through server-side sanitization (`sanitize-html`). Never use `dangerouslySetInnerHTML` on unsanitized content.

```typescript
// Correct pattern
import sanitizeHtml from 'sanitize-html'

const safeHtml = sanitizeHtml(content, { allowedTags: [...], allowedAttributes: {...} })

return <div dangerouslySetInnerHTML={{ __html: safeHtml }} />
```

### 8.5 Robots and Indexing

All shared agenda pages set `robots: noindex, nofollow`. These are private client documents and must not appear in search engine results. The `generateMetadata` function enforces this.

## 9. Routing and Middleware Integration

### 9.1 Route Group

If the app uses Next.js route groups (e.g. `app/(dashboard)/` for auth-protected routes), the `/shared/[token]/` route must be placed **outside** any auth-protected route group:

```
app/
├── (dashboard)/          # Auth-protected routes — Feature 24 middleware applies
│   ├── page.tsx          # Dashboard
│   └── agendas/
│       └── [short_id]/
│           └── page.tsx  # Agenda editor
└── shared/               # Public routes — no auth middleware
    └── [token]/
        └── page.tsx      # Shared agenda (this feature)
```

### 9.2 No Redirect on Unauthenticated Access

Auth middleware (Feature 24) must not redirect `/shared/*` routes to login. If middleware is implemented before this feature, add an exclusion for `/shared/*` immediately.

## 12. Deployment Notes

- The `/shared/*` route is publicly accessible — it must be reachable without VPN, auth cookies, or session state
- DNS routing: `ui.domain.com/shared/*` routes to the UI container (Next.js serves the page), which then calls `api.domain.com/shared/{token}` server-side
- The UI's server-side fetch to the API uses the internal API URL (from env var, e.g. `INTERNAL_API_URL`) — not the public-facing API domain — to avoid unnecessary DNS roundtrips in the container network
- Environment variables needed:
  - `INTERNAL_API_URL` — Internal base URL for server-side API calls (e.g. `http://api:3000`)
  - `NEXT_PUBLIC_API_URL` — Public API URL (not needed for this feature since all calls are server-side)
