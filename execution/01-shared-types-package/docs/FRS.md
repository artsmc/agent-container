# Functional Requirement Specification
# Feature 01: shared-types-package

## 1. Package Configuration

### FR-01: Package Identity

The package must be configured as an Nx library under the name `shared-types` with the npm package name `@iexcel/shared-types`.

**Requirements:**
- `project.json` must declare `"name": "shared-types"` and `"projectType": "library"`
- `package.json` must declare `"name": "@iexcel/shared-types"` with no `main` entry point (types-only package)
- `tsconfig.json` must extend the root `tsconfig.base.json`
- `tsconfig.lib.json` must configure the TypeScript compiler to emit declaration files (`.d.ts`) only, with `"emitDeclarationOnly": true` and `"declaration": true`
- The root `tsconfig.base.json` must include a path alias: `"@iexcel/shared-types": ["packages/shared-types/src/index.ts"]`
- Nx build target must use `@nx/js:tsc` executor

### FR-02: TypeScript Compiler Settings

The package must compile under strict TypeScript settings.

**Requirements:**
- `"strict": true` — enables all strict type checks
- `"noImplicitAny": true` — no implicit any types
- `"strictNullChecks": true` — null and undefined handled explicitly
- `"noUnusedLocals": true` — no unused type definitions
- TypeScript version must match the workspace root version (as defined in `package.json` devDependencies)

## 2. Module: task.ts

### FR-10: ShortId Branded Type

A branded string type must be defined to distinguish task short IDs from plain strings.

```typescript
type ShortId = string & { readonly __brand: 'ShortId' };
```

**Requirements:**
- Must be a nominal/branded type, not a plain `string` alias
- Format constraint documentation: values must match the pattern `TSK-NNNN` (e.g., `TSK-0001`, `TSK-0042`)
- Format validation is NOT enforced at the type level (validation is the API's responsibility); the brand exists to prevent accidental string assignment
- Must be exported

### FR-11: TaskStatus Enum

```typescript
enum TaskStatus {
  Draft = 'draft',
  Approved = 'approved',
  Rejected = 'rejected',
  Pushed = 'pushed',
  Completed = 'completed',
}
```

**Requirements:**
- Must use string values matching the database ENUM values exactly
- `Completed` — only set by historical import (Feature 38) or reconciliation cache writes (Feature 13); never set directly by user actions in the normal task lifecycle
- The design decision for `completed` usage constraints must be documented in a JSDoc comment on the enum
- Must be exported

### FR-14: ExternalRef Interface

```typescript
interface ExternalRef {
  system: string;
  externalId: string | null;
  externalUrl: string | null;
  projectId: string | null;
  workspaceId: string | null;
}
```

**Requirements:**
- `system` — identifies the PM tool (e.g., `"asana"`, `"monday"`, `"jira"`); required, never null
- `externalId` — the task ID in the external system; null until the task has been pushed
- `externalUrl` — deep link to the task in the external system; null until pushed
- `projectId` — the project within the workspace; null if not yet assigned
- `workspaceId` — the workspace/organization in the external system; null if not yet assigned
- This interface replaces the Asana-specific fields (`asana_task_id`, `asana_workspace_id`, `asana_project_id`) from the database schema, making the system PM-tool-agnostic
- JSDoc comment must explain the external_ref pattern and why it replaces Asana-specific fields
- Must be exported

### FR-15: TaskDescription Interface

```typescript
interface TaskDescription {
  taskContext: string;
  additionalContext: string;
  requirements: string[];
}
```

**Requirements:**
- Represents the structured description format generated by the Mastra agent
- Stored as JSONB in the database (JSONB-compatible structure)
- `taskContext` — background and context for the task
- `additionalContext` — supplementary context from the transcript
- `requirements` — array of specific deliverables or acceptance criteria for the task
- All fields required (not optional) — agent must always populate all three sections
- Must be exported

### FR-16: NormalizedTask Interface

```typescript
interface NormalizedTask {
  id: string;
  shortId: ShortId;
  clientId: string;
  transcriptId: string | null;
  status: TaskStatus;
  title: string;
  description: TaskDescription;
  assignee: string | null;
  priority: TaskPriority;
  estimatedTime: string | null;
  dueDate: string | null;
  scrumStage: string;
  tags: string[];
  externalRef: ExternalRef | null;
  approvedBy: string | null;
  approvedAt: string | null;
  pushedAt: string | null;
  createdAt: string;
  updatedAt: string;
}
```

**Requirements:**
- `id` — internal UUID (string representation)
- `shortId` — branded `ShortId` type (e.g., `TSK-0001`)
- `transcriptId` — nullable; some tasks may be created manually without a source transcript
- `estimatedTime` — ISO 8601 duration string (e.g., `"PT2H30M"` for 2 hours 30 minutes); nullable if not estimated
- `dueDate` — ISO 8601 date string (e.g., `"2026-03-15"`); nullable
- `scrumStage` — free-form string; default value is `"Backlog"` (enforced by API, not the type)
- `tags` — array of strings; empty array `[]` if no tags
- `externalRef` — null until the task has been pushed to an external system; populated after push
- All timestamp fields (`approvedAt`, `pushedAt`, `createdAt`, `updatedAt`) are ISO 8601 strings; approval/push timestamps are nullable until the event occurs
- Must be exported

  source: TranscriptSource;
  sourceId: string;
  meetingDate: string;
  clientId: string;

  durationSeconds: number;
  segments: TranscriptSegment[];

## 9. Workflow and Interaction Patterns

### FR-80: Consumer Import Pattern

All applications and packages in the monorepo must import from the package name, not from relative paths:

```typescript
// Correct
import { NormalizedTask, TaskStatus } from '@iexcel/shared-types';

// Incorrect — relative paths break if the consumer moves
import { NormalizedTask } from '../../packages/shared-types/src/task';
```

The `@iexcel/shared-types` path alias in `tsconfig.base.json` enables this.

### FR-81: Type-Only Imports

Consumers should prefer `import type` syntax to make it explicit that no runtime values are imported:

```typescript
import type { NormalizedTask, TaskStatus } from '@iexcel/shared-types';
```

The package itself must not export any runtime values, so both `import` and `import type` must produce identical results.

### FR-82: No Circular Dependencies

The `shared-types` package must have zero dependencies on any other package in the monorepo. It is the root of the dependency graph.

## 10. Error Handling and Edge Cases

### FR-90: Optional vs Required Field Strictness

- Fields that the database allows to be NULL must be typed as `T | null` in interfaces (not `T | undefined` or just `T`)
- Fields that may be absent in request payloads must be typed as `field?: T` (optional property with `undefined`)
- The distinction between `null` (explicitly absent) and `undefined` (not provided) must be respected

### FR-91: Enum vs Union Type Decision Rule

| Use Enum When | Use Union Type When |
|---|---|
| Values map to database ENUM values | Values are purely TypeScript-level constructs |
| The set of values is stable and closed | The set may expand with new integrations |
| Strong nominal typing is desired | Lightweight inline values are sufficient |

Applied in this package: `TaskStatus`, `AgendaStatus`, `TaskSource`, `TaskPriority`, `UserRole`, `MeetingType`, `ApiErrorCode` are enums. `TranscriptSource` and `WorkflowStatus` are union types.
